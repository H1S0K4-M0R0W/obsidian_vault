# - Создание коммита

> [!cite] **`Commit`** (фиксировать) - 
> Это снимок (`snapshot`) состояния проекта в определенный момент. Он фиксирует изменения, внесенный в файлы репозитория, сохраняет их в истории проекта.

> [!success] Основные команды для отслеживая `snapshot`
> - `git status` - Вывод информации о состоянии **каждого файла Git-репозитория**
> - `git log` - Вывод информации о каждом существующем `commit`

Пример работы программы `git status`:

``` shell
git status

> Текущая ветка: main
Еще нет коммитов
нечего коммитить (создайте/скопируйте файлы, затем запустите
«git add», чтобы отслеживать их)

touch file.txt
echo "Hello W0rld" >> file.txt
git status

> Текущая ветка: main
Еще нет коммитов
Неотслеживаемые файлы:
  (используйте «git add <файл>...», чтобы добавить в то, что будет включено в коммит)
 --> file.txt
индекс пуст, но есть неотслеживаемые файлы
(используйте «git add», чтобы проиндексировать их)

git add file.txt # Добавили отслеживание файла в Git-репозиторий
git status

> Текущая ветка: main
Еще нет коммитов
Изменения, которые будут включены в коммит:
(используйте «git rm --cached <файл>...», чтобы убрать из индекса)
новый файл:    file.txt

```

Все файлы созданы, подготовлены и отслеживаются, теперь можно зафиксировать (`за-commit-ить`) состоянии репозитория
Здесь существует несколько способов:

``` shell
1. git commit
   # Затем откроется редактор в терминале и можно будет вписать текст *комментарий* для данной фиксации *коммита*

2. git commit -m "Comment"
   # Вписать коментарий самостоятельно: -m (message - "сообщение")
```

> [!tip] Кстати...
> Для новичков редактор `Vim` может быть немного сложен в использовании, поэтому рекомендую использовать редактор `Nano`

> [!warning] Осторожно, Vim!
> Если на этом моменте Вас `Vim` застал врасплох то следуйте командам:
> 1. Нажмите клавишу `I` - сопровождается появлением надписи `~INSERT~`
> 2. Вписывает Ваш коментарий
> 3. Выходите из режима редактирования - нажатие клавиши `Esc (Escape)`
> 4. Сохраняем изменения (Ваш комментарий НЕ ДОЛЖЕН начинаться с `#` - если же это не так, удалите символ `#`) - введите `:wq` (`: write quit`)
> 5. Результат:
> ``` shell
> git commit
[main (корневой коммит) 8852dd5] add file file.txt
 1 file changed, 1 insertion(+)
 create mode 100644 file.txt
> ```

# - Что такое индекс (`Stage`)?

![[Structure of Stage.png]]
1. Рабочая директория - файловая система
2. `Stage` - изменения в файлах
3. Репозиторий - Список все версий

> [!success] Структура взаимосвязи команд в `Git`
> `Working Directory` -> *git add* -> `Stage`
> `Stage` -> *git commit* -> `Repository`

> [!success] Переходные состояния файлов в`Git`
> `Untracked` -> *git add* -> `Staged`
> `Staged` -> *git commit* -> `Unmodified`
> `Unmodified` -> *edit* -> `Modified`
> `Modified` -> *git add* -> `Staged`
> `Unmodified` -> *remove* -> `Untracked`
> 
> ![[File State.png]]

> [!warning] Важно о коммитах
> Если Вы вносите изменения после выполнения команды `git add` то они не будут добавлены в Ваш репозиторий, нужно будет ввести команду в терминале ещё раз!

Для закрепления понимания данной специфической особенности системы контроля версий проведем небольшую практику:
1. Создадим файл и внесем в него изменения
2. Проиндексируем его
3. Еще раз внесем изменения и *НЕ БУДЕМ ИХ ИНДЕКСИРОВАТЬ!*
4. Сделаем коммит
5. Удалим файл 
6. Сделаем откат на предыдущий коммит
7. Посмотрим содержимое файла
==Опционально:==
8. Если Вы хотите видеть состояния, то можно пользоваться командой `git status` 
Приступим:
``` shell
mkdir git-example
cd git-example/

touch file.txt
echo "Hell0 W0rld" >> file.txt

git add file.txt

echo "Hi, AdMiN" >> file.txt

git commit -m "edited file.txt"

git log --oneline

rm file.txt

# Отменить все изменения после последнего коммита
# (подробнее будет рассмотрена далее)
git reset --hard

cat file.txt

> "Hell0 W0rld"
# Вывод - второй строки нет!

```

# - *Ленивый* коммит

Если вы хотите сократить выполнение 2 команд, а именно `git add . + git commit` в 1 используйте данную команду
```shell
# -a означает все файлы (--all)
git commit -a

# Подробный просмотр  последнего успешного коммита
git show

# или же просмотр ЛЮБОГО коммита путем указания его ID
# Пример:
git log --oneline
90edf00 (HEAD -> main) edited file.txt
ae213a9 remove file.txt
8852dd5 add file file.txt

git show 90edf00

commit 90edf009d468bad74528d0a638d4a3f0cb0a3446 (HEAD -> main)

    edited file.txt

diff --git a/file.txt b/file.txt
new file mode 100644
index 0000000..557db03
--- /dev/null
+++ b/file.txt
@@ -0,0 +1 @@
+Hello World

# Узнать только имена файлов
git show --name-only <ID>

```

> [!warning] Важное уточнение для `git commit -a`
>  Команда `git commit -a` считывает предыдущие коммиты и файлы, которые были в них, поэтому при создании файла, которого ранее не было, стоит использовать *классическую связку* `git add . + git commit` 
>  
>  Кстати можно делать еще более ленивые сочетания:
>  `git commit -am "comment"`
>  
> Если вы уже столкнулись с данной проблемой (см. ранее) то смотрите код ниже:

Для того чтобы добавить файл в предыдущий коммит необходимо воспользоватся командой:

```shell

git add .
git commit --amend
# --amend - внести поп равки в предыдущий коммит
# amend - (amendment) - поправка, изменения*
 
```

> [!success] Уточнение для `git commit`
> При использовании команды `git commit` появляется терминальный текстовый редактор и там можно сделать много строчный коммит который будет по-разному отображаться
> Например:
> ``` shell
> git commit
> 
> edited file.txt
> <!--- пропускаем данному строку -->
> we wrote Hell0 W0lrd
> ```
> При использовании `git log` мы увидим комментарий полностью, но с дополнительным параметром `--oneline` мы увидим только `edited file.txt`

> [!example] Правила хорошего коммита:
> 1. Пустая строка между темой и телом
> 2. Тема сообщения не более 50 символов
> 3. (Часто) Тема в верхнем регистре (CapsLock)
> 4. Точка НЕ СТАВИТСЯ в коне темы
> 5. Тело не более 80 символов
> 6. Тело объясняет что изменено
> 7. Тема отвечает на вопрос: **Что делает коммит?"**

# - Сброс индекса

Существует несколько сценариев использования команды `git reset` , а именно:
1. Сброс команды `git add` - например Вы добавили лишний файл и не хотите его присоединять к коммиту (`git reset`) \*также это можно сделать через данную команду - `git reset <filename>`\*
2. Возврат к предыдущему коммиту - например Ваш новый коммит сломал проект и Вы хотите сделать откат к предыдущей версии `git reset --hard` НО такое использование удаляет изменения без какой-либо регистрации
3. Возврат к определённому коммиту без удаления текущего - `git reset HEAD~X`, где Х - это число шагов назад.
	**Важно!** Если вы хотите вернутся на первоначальный коммит Вам необходимо прописать команду, в которой будет указан UID коммита: `git reset 0a2b3c4d`
	**Помните!** Когда вы используете команду `git reset HEAD~X` первоначальный коммит перестает отображаться в `git log` поэтому сохраните UID заранее!
	Если Вы вызывали команду `git reset HEAD~X` или `git reset^` - аналогично `git reset HEAD~1` не более 1 раза, то еще не всё потеряно, тк `Git` сохраняет ранее использованный UID:
	**Для получения UID** воспользуйтесь командой: `cat .git/ORIG_HEAD` или же `git reset ORIG_HEAD`

	Подробнее о ключе `--hard`:  он не сохраняет изменения в директории!

> [!succes] Как работают параметры для `git reset`
> 1. `--hard`
> 2. `--soft`
> 3. `--mixed`
> ![[git reset --param.png]]
> 
> Подробнее:
> `--soft` - затрагивает только изменения в репозитории но не из коммита и рабочей папки
> `--mixed` - затрагивает изменения в коммите и репозитории но не в рабочей папке (по-умолчанию)
> `--hard` - затрагивает все изменения 
> Если хотите больше понимать принцип работы `--soft` то это можно интерпретировать как:
>  `git reset --soft HEAD~1 = git commit --amend` 
>  

Проще говоря

| Параметр          | `--soft` | `--mixed` | `--hard`                                              |
| ----------------- | -------- | --------- | ----------------------------------------------------- |
| `git commit`      | +        | +         | +                                                     |
| `git add .`       | -        | +         | +                                                     |
| Working Directory | -        | -         | + (полная перезапись до состояния выбранного коммита) |

# история коммитов в кратком виде
git log --oneline

git show

# откатить последний коммит без ввода сообщения для коммита 
git revert HEAD --no-edit

git revert 8ab14c 
 
# откатить третий коммит с конца 
git revert HEAD~3 

# отменить незавершенный откат коммита 
git revert --abort 
 
# откатить серию коммитов 
git revert 9e59768..4b2413e 
 
# откатить коммит без создания нового коммита (изменения попадают в индекс)
git revert 9e59768 --no-commit

# - Удаление файлов

> [!warning] Как так то...
> Так как при использования `Git` файлы поступают в коммит, удаление данного файла переведет его в статус `Untracked` -> после Вам придется выполнить `git add .`, тогда файл станет `Staged` и лишь затем `git commit` чтобы окончательно и бесповоротно его удалить из коммита
> Но есть более простое и лаконичное решение а именно: `git rm <filename>`
> Он сразу переводит файл в состояние `Staged` и его можно сразу же коммитить

Если же файл был закоммичен, то необходимо добавить параметр `-f` (`--force`) чтобы его удалить из `Staged`
При удалении папок используйте параметр `-r` (`--recursive`) для удаления папки с её содержимым

> [!cite] `git clean`
> - Удаление файлов, находящихся в состоянии `Untracked`
> - По-умолчанию необходим параметр `--force` (`-f`)
> - Для удаления директорий необходим параметр `-d`

```shell
# вызов справки 
git clean --help 

# сценарий 1 
# создадим ненужные файлы
touch some_binary_file{1..9}.bin 
ls
# посмотрим их статус git status 
# очистим Git от этих файлов
git clean -f
ls
git status

# сценарий 2
# создадим пустые папки
mkdir bin
mkdir tmp
ls
# посмотрим их статус
git status 
# очистим Git от этих пустых каталогов 
git clean -d -f 
ls 
git status 
# ключ -n -- 'холостой прогон команды' 
# покажет, что будет происходить при выполнении команды 
git clean -f -n 
git clean -f -d -n 
# ключ -q -- тихое выполнение команды # не выводит комментарии о ходе своего выполнения git clean -f -d -q
```

# - Переименование файлов (`git mv`)

> [!cite] `git mv` (move - перемещать)
> Данная команда позволяет переименовывать файлы, а также перемещать их как `Вырезать -> Вставить`
> Чтобы понять сколько экономии времени дает данная команда, можно пройти данную операцию через команду `mv` - базовая утилита для любого `Linux`
> 1. `mv <filename1> <filename2>` -> filename1 - `Untracked`, filename2 - `Untracked`
> 2. `git rm <filename1>` -> filename1 -  `Removed`, filename2 - `Staged`
> 3. `git commit` -> filename2 - `Unmodified`
> Очень кропотливо и долго!

> [!success] Иначе...
> А с использованием `git mv <filename1> <filename2>` вы сразу получаете `<filename2>` в `Staged` и просто его коммитите
> Вот так, легко и просто!

```shell
# переименование файла в терминале 
mv page1.html index.html 
# набор команд для корректного переименования в Git 
mv page1.html index.html 
git add index.html 
git rm page1.html 

# переименование/перемещение файла в Git одной командой 
git mv page1.html index.html 
# вызов справки git mv --help # создаем папку styles 
mkdir styles 
# перемещаем файл с CSS-стилями в новую папку 
git mv styles.css styles 
# делаем коммит 
git commit -m "create folder for css styles" 
# перименование папки в Git 
git mv styles css 
# создаем папку web_pages 
mkdir web_pages 
# перемещение набора файлов 
git mv *.html web_pages
```

# - Просмотр изменений в `Git` (`git diff`)

> [!cite] `git diff` (`differences` - различия)
> Именно это команда в `Git` позволяет просматривать различия в файлах, если Вы хотите понимать что было изменено в проекте

Данная утилита `Git` позволяет не только смотреть разницу в файлах и коммитах, но с помощью другой программы просматривать файлы ВНЕ репозитория, называется она `git difftool`

Для Вашего удобства изменения открываются последовательно и в терминальном текстовом редакторе. Круто!

![[git difftool.png]]

```shell
# вызов локальной справки по команде 
git diff 
git diff --help 
# вывод изменений после последнего коммита (не добавленных в индекс) 
git diff 
# вывод изменений после последнего коммита (включая добавленные в индекс) 
git diff HEAD 
# вывод изменений, добавленных в индекс 
git diff --staged 
git diff --cached 
# фоматирование изменений по словам (а не по строкам) 
git diff --word-diff 
# исключение пустых строк из вывода 
git diff -w 
# вывод изменений после HEAD~1 (и до HEAD) 
git diff HEAD~1 
# указание хэша коммита для вывода изменений после него 
git diff 8a8b14c 
# вывод разницы между двумя указанными коммитами 
git diff 8a8b14c b0272be 
# вывод изменений в конкретном файле после указанного коммита 
git diff HEAD~1 web_pages/index.html 
# сравнение двух произвольных файлов (даже вне git-репозитория) 
git diff index.html page2.html 
# вызов локальной справки по команде 
git difftool 
git difftool --help 
# указание хэша коммита для вывода изменений после него с помощью утилиты difftool 
git difftool 8a8b14c
```

Также для просмотра изменений можно использовать `git show`, работают он похоже но РАЗНИЦА ЕСТЬ и её важно понимать:
1. Если вы воспользуетесь командой `git show` ты Вы дополнительно увидите ещё и принадлежащий индексу комментарий, также изменения будут разделены на отдельные коммиты
2. Если вызвать те же  самые коммиты в `git diff`, то утилита отобразит все изменения вместе

==`git diff HEAD filename2`== vs ==`git show HEAD:filename2`==

Если Вы воспользуетесь параметром `--name-only` то результаты будут совсем разные, например:
`git diff --name-only HEAD`
 - Вы увидите все файлы изменённые с первого по текущий коммит
 `git show --name-only HEAD`
  - Вы увидите изменения только в текущем коммите

``` shell
# просмотр коммита (одно и то же) 
git show git diff HEAD~1 
# вывести все накопленные изменения между текущим состоянием и состоянием на момент указанного коммита 
git diff bea5ea0 
# вывести только изменения, выполненные в указанном коммите 
git show bea5ea0 
# вывести все коммиты с изменениями между двумя коммитами 
git show 0e47d61..ca3a6b1 
# вывести все накопленные изменения в файлах между двумя коммитами 
git diff 0e47d61..ca3a6b1 
git diff 0e47d61 ca3a6b1 
# аналогичные команды - просмотр изменений файла в определённом коммите 
git show HEAD~1:styles/styles.css 
git diff HEAD~1 css/styles.css 
# вывести только список файлов, измененных в указанном коммите 
git show 4b2413e --name-only 
git show 55f77dc --name-only 
# вывести только список файлов, измененных с указанного коммита и до текущего состояния репозитория 
git diff 55f77dc --name-only
```

# - История коммитов (`git log`)

Изучим новые параметры для более подробного изучения коммитов

`git log --stat` - позволяет показать где происходили изменения, и сколько всего файлов было задействовано в коммите

`git log -p` (`--patch` - заплатка) - позволяет подробнее изучить изменения, производимые над каждым файлом

`git log -2` - позволяет посмотреть подробно `2` коммита (`-2`)

`git log --graph` - позволяет вывести коммиты в псевдографическом последовательном формате

`git log --author="Author"` - если у Вас есть несколько веток, то можно посмотреть вклад другого пользователя (автора)

`git log --pretty=format:"%h - %an - %as - %s"` - это форматируемый вывод (%буквы смотреть в документации) `%h` - hash-код, `%an` - автор, `%as` - время, `%s` - сообщение
(`%n` = `\n` = перенос строки)

Также есть еще и `git shortlog` - данная команда выводит информации о каждом авторе и созданными ими коммитами (`-e` `--email` - вывод электронной почты авторов)
Параметры:
`-n`, `--number` - вывод количества коммитов,
`-s`, `--summary` - суммирует информацию о коммитах авторов
`--since="2026-XX-XX"` - указать дату с которой нужно сделать выборку коммитов
`--until="2026-YY-YY"` - обратная, параметру `--since=""`

``` shell
# базовое использование команды просмотра логов
git log

# просмотр логов в кратком формате
git log --oneline

# вызов справки по команде
git log --help

# вывод статистики по коммитам
git log --stat	

# просмотр патчей (деталей каждого изменения) по коммитам
git log -p

# ограничение количества комитов в выводе
git log -4

# опции можно сочетать в одной команде
git log –-stat -p
git log -4 --stat -p

# отображение веток в истории коммитов
git log --graph

# клонирование нового репозитория в отдельной папке
cd ..
git clone https://github.com/githubtraining/hellogitworld
cd hellogitworld
ls –a

# использование изученных команд в новом репозитории
git status
git log
git log --oneline
git log --oneline README.txt
git log --stat	
git log --graph

# фильтр коммитов по автору
git log --author="Jordan McCullough" –-oneline

# форматирование: дата, автор, время, сообщение
git log -5 --pretty=format:"%h - %an - %as - %s"
git log -5 --pretty=format:"%h - %an - %as %n >> %s"

# читабельное предложение с помощью форматирования
git log --pretty=format:"This guy:%cn committed with hash '%h' on %cd"


# компактный отчёт о коммитах в репозитории группирует коммиты по авторам и выводит список их имён вместе с количеством сделанных ими коммитов и краткими описаниями этих коммитов
git shortlog

# показывает электронную почту автора рядом с его именем
git shortlog -e

# сортирует вывод по количеству коммитов, от большего к меньшему
git shortlog -n

# выводит только статистику (количество коммитов) без списка сообщений коммитов
git shortlog -s

# фильтрация коммитов по датам "до" и "после"
git shortlog --since="2012-01-01" --until="2012-12-31"

```

Полезные опции для `git log --pretty=format`

| Опция | Описания вывода                                             |
| ----- | ----------------------------------------------------------- |
| `%H`  | Хеш коммита                                                 |
| `%h`  | Сокращённый хеш коммита                                     |
| `%T`  | Хеш дерева                                                  |
| `%t`  | Сокращённый хеш дерева                                      |
| `%P`  | Хеш родителей                                               |
| `%p`  | Сокращённый хеш родителей                                   |
| `%an` | Имя автора                                                  |
| `%ae` | Электронная почта автора                                    |
| `%ad` | Дата автора (формат даты можно задать опцией --date=option) |
| `%ar` | Относительная дата автора                                   |
| `%cn` | Имя коммитера                                               |
| `%ce` | Электронная почта коммитера                                 |
| `%cd` | Дата коммитера                                              |
| `%cr` | Относительная дата коммитера                                |
| `%s`  | Содержание                                                  |
